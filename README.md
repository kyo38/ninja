SCIONからNinjaへ
Rustによる次世代セキュア経路制御基盤
開発ロードマップ（完全版）

==================================================

本書は、Path-Aware Networkingの思想を基盤として、
Rustによる次世代セキュア経路制御基盤を構築するための
開発指針をまとめたものである。

==================================================
■ システム構成
==================================================

 Master（Control Plane）
 Worker（Execution Node / Data Plane）
 Client（Trigger）

 DAG（依存関係グラフ）
 Event-driven実行

--------------------------------------------------

        +---------+
        | Client  |
        +----+----+
             |
             v
        +---------+      TCP (9001)
        | Master  |<------------------+
        +----+----+                   |
             |                        |
             | TCP (9000)             |
             v                        |
        +---------+                   |
        | Worker  |-------------------+
        +---------+

--------------------------------------------------

[制御フロー]
Client -> Master -> Worker

[状態同期]
Worker -> Master (Notify)

==================================================
■ 実験手順（完全版）
==================================================

本分散システムのテストを行うため、
VSCodeのターミナル（PowerShell等）を「3つ」開く。

----------------------------------------
1. セットアップ
----------------------------------------
git clone https://github.com/kyo38/ninja
cd ninja
cargo check

----------------------------------------
2. Master起動
----------------------------------------
cargo run --bin ninja

ポート9001（対Worker）および9000（対Client）で待機状態となる。

----------------------------------------
3. Worker起動
----------------------------------------
cargo run --bin worker

起動後、自動的にMasterへソケット接続し、
クラスタへチェックインして待機状態となる。

----------------------------------------
4. Client起動
----------------------------------------
cargo run --bin client

4つのタスクを含むJSONをMasterへ送信し、
Clientは即座に終了する。

----------------------------------------
■ タスク実行検証
----------------------------------------

以下の順序保証を確認する：

A の受信・実行・完了報告
B の受信・実行・完了報告
C（Aに依存）の受信・実行・完了報告
D（最終リーフ）の受信・実行・完了報告

→ DAG依存順序が厳密に守られること

----------------------------------------
■ 並列実行検証
----------------------------------------

さらに「4つ目のターミナル」を起動し：

cargo run --bin worker

Workerを追加する。

その後Clientを再実行すると：

- A と B が別Workerへ分配される
- 並列実行される
- Masterが動的にスケジューリングする

→ 並列負荷分散を確認可能

==================================================
■ エラー分類
==================================================

Success     : 正常終了
Failure     : 論理的失敗（リトライ対象）
InfraError  : インフラ障害（リトライ対象）

→ 分離によりリトライ戦略を制御可能

==================================================
■ 初期バグと修正
==================================================

問題：

- Worker実行中に次タスクが送信される
- DAG依存を無視したフライング実行

原因：

- 非同期処理で完了待機していなかった

修正：

- state_mapによる状態管理
- Notifyによる完了同期

結果：

- DAG順序保証が実現
